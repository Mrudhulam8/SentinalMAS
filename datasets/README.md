## Large datasets (scale / performance demos)

Generated by [`generate_datasets.py`](generate_datasets.py). Each of the 20
attacker IPs is restricted to a fixed subset of attack kinds ("profile"), so
incidents land across a genuine spread of risk scores/threat levels instead
of every IP maxing out the risk formula (severity=high + diversity capped at
4 + reputation always converges to risk 10.0/Critical for everyone if every
IP can trigger every kind — that's what earlier versions of this generator
did, producing a flat "all Critical" risk chart).

| File | Format | Logs | Incidents | Threat levels | Distinct risk scores |
|---|---|---|---|---|---|
| `large_logs_10k.csv` | CSV | 10,000 | 20 | 10 Critical / 6 High / 4 Medium | 5.6 – 10.0 (6 values) |
| `large_apache_access_10k.log` | Apache | 10,000 | 20 | 10 Critical / 4 High / 6 Medium | 5.6 – 10.0 (5 values) |
| `large_linux_auth_10k.log` | Linux auth | 10,000 | 8 | 7 High / 1 Critical | 7.6 – 8.6 |
| `large_logs_50k.csv` | CSV | 50,000 | 20 | 10 Critical / 6 High / 4 Medium | 5.6 – 10.0 (6 values) |

The Linux auth log only has fewer incidents by design — an SSH auth log can
only naturally represent brute-force login attempts, so only the attacker IPs
whose profile includes `brute_force` produce a real signal there; the rest
are format-appropriate, not a bug. `large_logs_10k.csv` is the best all-round
demo — every detection rule fires and the risk chart shows real variation.
The full pipeline processes 10k logs in well under 1s and 50k in ~2s
(keyless); see [../docs/testing-and-performance.md](../docs/testing-and-performance.md).

## Regenerating / custom sizes

```bash
# Regenerate the default set (the four files above)
python datasets/generate_datasets.py

# Custom size / format
python datasets/generate_datasets.py --count 100000 --format csv   --out datasets/huge.csv
python datasets/generate_datasets.py --count 25000  --format apache --out datasets/web_25k.log
```

Attacker IPs use the reserved TEST-NET ranges (RFC 5737), so they are safe to
publish and never touch real hosts.

## Note on live threat-intel keys

Detection is keyless and instant. When AbuseIPDB / VirusTotal keys are
configured, the Threat Intel agent looks up reputation **per unique IP**
(cached, and fetched concurrently per IP) — but those free tiers are
rate-limited (VirusTotal: 4 requests/min), so a large upload can contain far
more unique attacker IPs than they can service on demand. Enrichment is
hard-bounded to at most 25 new IP lookups and 8 seconds of wall time per
pipeline run, so a large batch can never stall the request — any IPs beyond
that budget simply proceed without live reputation data (MITRE mapping still
applies to every finding regardless). These datasets additionally use a small
pool (~20 attacker IPs) so a single run can fully enrich within the budget.
